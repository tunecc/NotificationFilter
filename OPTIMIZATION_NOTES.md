# Optimization Notes

本文件只做记录，不代表这些优化已经实施。

目的：

- 记录 `NotificationFilter` 后续可能继续做的轻量化方向
- 说明每个方向的收益、风险、优先级
- 避免以后重复讨论同一轮取舍

---

## 当前状态

截至当前记录时：

- 已经完成 `arm64e only`
- hook 逻辑未改
- 运行路径仍然是 `NCNotificationDispatcher::postNotificationWithRequest:`

当前关键事实：

1. 只注入 `SpringBoard`
2. 只在 `com.apple.shortcuts` 下做过滤
3. 当前匹配条件是：
   - 标题 `Automation failed`
   - 正文包含 `Remote execution timed out`

---

## 优先级排序

### P1. 只保留 `arm64e` slice

状态：

- 已完成

收益：

- 对包体和 dylib 体积有明确下降
- 不改变运行逻辑

本轮实际结果：

- dylib 从多架构版本约 `166832` bytes 降到单架构约 `68528` bytes
- 正式 `.deb` 从约 `5014` bytes 降到约 `3938` bytes

风险：

- 兼容范围收缩到 `arm64e`

为什么排第一：

- 收益最直接
- 风险最低
- 不涉及私有调用链重新判断

---

### P2. 热路径瘦身

状态：

- 未实施，只做记录

目标：

- 小幅降低运行时判断成本
- 让 `__text` 再小一点

当前可见的瘦身点：

1. 去掉通用 helper
   - `NFCallObject`
   - `NFNormalizedStringValue`
   - `NFSectionIdentifierCandidate`

2. 少用运行时动态拼装
   - 少用 `NSArray`
   - 少用 `NSSelectorFromString`

3. 改成固定 fast path
   - 先直接读最可能命中的字段
   - 只有失败时才走兼容回退

潜在改造方向：

- 预先定义静态 `SEL`
- 改成更直接的 `objc_msgSend` 分支
- 把标题/正文读取路径拆成明确层级，而不是循环尝试

收益判断：

- 运行时收益：小
- 二进制收益：小
- 可维护性影响：可控

风险：

- 低
- 主要风险是把代码写得过于底层后，可读性下降

为什么排第二：

- 不需要重新逆向
- 改动安全
- 虽然收益有限，但基本不会把系统行为搞坏

---

### P3. 换到更窄的 hook 点

状态：

- 未实施，只做记录

目标：

- 进一步降低热路径成本
- 让无关通知根本不进入我们的过滤逻辑

当前 hook 点：

- `NCNotificationDispatcher`
- `postNotificationWithRequest:`

这属于较靠后的全局通知派发链路。

理论上的更优方向：

- 找到只在 `Shortcuts` 自动化失败通知生成时才触发的方法
- 或者找到更靠近 `com.apple.shortcuts` 内容构造阶段、但不会影响全局通知派发的方法

潜在收益：

- 运行时成本上限最好
- 无关通知完全不进我们的判断代码

主要风险：

1. 需要重新逆向
2. 私有实现更脆
3. iOS 升级后更容易失效
4. 一旦选错点，维护成本会明显上升

为什么排第三：

- 上限高
- 但研发成本和失效风险都明显大于热路径瘦身

适用前提：

- 只有在你后续真的很在意这点热路径开销，或者继续做更复杂的通知过滤时，才值得继续挖

---

### P4. 改成手写 `MSHookMessageEx` / 去 Logos / 去 ARC / 强行去 Foundation

状态：

- 明确不优先

目标：

- 进一步压缩“源码层面的样子”

预期收益：

- 通常很小
- 更多是“看起来更底层”，不一定带来可感知收益

为什么收益小：

1. Logos 本质是编译前预处理器
2. 当前这类 tweak 的主要成本不在 Logos 语法本身
3. 现有工程已经非常小，继续压缩基础框架收益有限

主要风险：

- 代码更难维护
- 更容易引入细小错误
- 后续回看和继续改动成本更高

为什么排第四：

- 性价比最低

---

## 结论

当前建议保持如下顺序：

1. `arm64e only`
2. 热路径瘦身
3. 更窄的 hook 点
4. 手写底层 hook / 去 Logos / 去 ARC

对这个项目来说，最值得继续做的不是“盲目更底层”，而是：

- 先把**架构切片**收紧
- 再把**热路径**写得更直接
- 最后再判断是否值得为了更小的运行时成本去重新逆向更窄的 hook 点
